# Workshop v0.1.8 release notes

## 31 January 2025

These release notes cover new features and changes in Workshop v0.1.8.

This is the first release to offer an expanded version of the notes. This format

## What’s New in Workshop v0.1.8

### A Command to List Workshop Scripts

For example:

```console
$ cat workshop.yaml

  name: ws
  base: ubuntu@24.04
  scripts:
    foo: echo foo
    bar: |
      echo bar

$ workshop scripts ws

  bar:
    script: |
      echo bar
  foo:
    script: echo foo
```

The response format includes `script:` so it can be extended without breaking
anything.

### The `read-only` Option for Mounts

This enables the enforcement of read-only access to Workshop mounts from within
a workshop.

**Syntax:**

```
plugs:
  example_plug:
    interface: mount
    workshop-target: <target>
    [read-only: <true, false, "true", "false">]
```

The `read-only` parameter accepts a boolean or any string representation thereof
(e.g. `"true"`, `"false"`, `"TRUE"`, etc.).

## Minor Features

### Mounting SDKs as Read-Only Volumes

Managing SDKs as read-only volumes is required to support restoring from a
previous Workshop snapshot for workshop refresh. An SDK is attached to a
workshop as a read-only volume before running other tasks.

One caveat to consider is that the system will have as many versions of an SDK
volume as there are revisions used by running workshops at any given time.
Without ZFS deduplication or compression, this could lead to unnecessary
duplication, assuming an average SDK does not have significant content
differences between its revisions. The possible duplication issue is outside the
scope of this release. Based on the assumed use cases, it can be tolerated or
mitigated with ZFS deduplication in future Workshop releases.

On another implementation note, Workshop must transform SDKs into an acceptable
LXD backup tar format to import them as volumes. This is a temporary solution
and requires an extension to the LXD API to allow importing an arbitrary tarball
into a custom volume without attaching it to a workshop or using the backup
functionality.

### A Separate LXD Bridge for Workshops

At present, this provides an additional level of separation between Workshop
containers and other LXD containers users might have. In the future, we may want
to lock down this interface (to prevent issues like
[moby/moby#14041](https://github.com/moby/moby/issues/14041)) or apply a
workaround for the [UFW
issue](https://documentation.ubuntu.com/lxd/latest/howto/network_bridge_firewalld/#ufw-add-rules-for-the-bridge).

### Code Coverage

- We now use the [TIOBE TICS framework](https://www.tiobe.com/products/tics/)
  for coverage and quality analysis.
- We have also enabled the `errcheck` linter for new code (triaging legacy code
  will take some time, though).
